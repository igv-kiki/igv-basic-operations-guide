iGV 基本盘维持运营手册

本手册回答四件事:用户为什么留在 iGV、维持基本盘要守什么标准、每天每周怎么执行、谁来负责。目标是把基本盘运营机制化,让团队从被动转为主动,管理者只看周报、只处理升级项。

监控范围

红线基准期
2025.01–2026.05
执行团队
1 负责人 + 2 卖家管理客服
PART 01
基本盘说明
先讲清楚用户为什么留下、我们要守什么、守住的价值是什么。结论先行。
1.1

用户为什么现在还留在 iGV

结论

在流量与流水均无增长的现状下,留下来的用户主要是金币(Coins)类的复购用户。他们留下的关键是三点:服务体验好、价格够低、所玩游戏(以 Dofus 为主)有持续热度。其中游戏热度不可控,而服务与低价是平台可控、并已经做好的两件事——这就是基本盘要守住的核心。

支撑结论的关键数据

42.9%
Coins 买家复购率(2026),远高于 Accounts 的 20%——服务好的品类,用户更愿意回来
58%
直接访问用户复购率,且逐年上升(49%→55%→58%)——靠信任主动回来的人在增加
2.36 次
Coins 用户人均复购次数——金币是高频补给型消费,复购是基本盘的支柱

三组数据指向同一结论:留存集中在服务做得好的高频品类,且越来越依赖"用户主动回来"而非买量。这说明沉淀下来的是信任,而信任来自服务与低价。服务侧的改善是实打实的——准时递送率 71%→92%、退款率 14%→3.7%、好评率 91%→96%(2024→2026),用户留存的改善正建立在这个基础上。

注:Dofus 单一游戏约占总 GMV 39%,热度由游戏方版本节奏驱动,平台不可控,只能追踪与承接(见 §2.3 / §3.3)。

1.2

维持基本盘的核心运营动作

对应"用户为什么留下",要守的就是可控的两件事,加两项辅助监测。判断要不要重点守,标准是:这件事停了,用户会不会走。

运营动作定位核心做法
服务履约维持核心设红线 + 趋势线,全业务监控、亮灯处置。服务做差用户一定走;且服务是平台级信任,任一业务出问题都影响用户对整个平台的信任。
低价竞争力监测核心监测已具备价格优势的指定业务,守住优势不滑坡。低价没了用户会走。
游戏动态追踪辅助不可控,只追踪,用于解释订单量变化。
复购监测辅助不设红线。复购是结果指标,异常时用来反查是需求还是服务出了问题。
1.3

守住基本盘的本质,是守住"用户为什么选 iGV"这件事本身。在此前提下,这套机制还带来三层管理价值:

PART 02
监测标准
这一部分只规定"判断好坏的尺子":指标、红线、阈值、分档。具体怎么操作见第三部分。
2.1

服务履约标准

2.1.1　指标分组

指标类型含义处置
好评率主要用户满意、愿意再来的直接结果硬红线 · 破线升级
递送完成率主要订单是否最终送达硬红线 · 破线升级
准时递送率主要是否在承诺时间内送达硬红线 · 破线升级
订单退款率主要按笔数,用户遇到问题的频率硬红线 · 破线升级
售后发起率次要用户主动发起售后的比例观察线 · 只提示
金额退款率次要按 GMV,退款的损失规模观察线 · 只提示

指标依据:四个主要指标覆盖履约全环节——能否送达、是否及时、是否满意、是否造成损失;次要指标补充售后频率与损失规模两个侧面。

2.1.2　三层结构

层级看什么报警逻辑
平台级全业务汇总 6 指标绝对红线
品类级ACC / Coins 各自 6 指标(其他品类暂不设)各自绝对红线 + 趋势线
业务级TOP10 业务、增长苗子各自业务自比趋势线为主 + 套所属品类红线
分品类设红线的原因

品类服务特性不同,绝对值不可横向比。Accounts 售后率天然偏高、好评率天然偏低(账号交易存在封号、找回等问题);Items 准时率偏低(未必准时,但最终能送达,且不会出现封号找回等问题,差评和争议概率低)。所以红线各设各的,判断好坏看各自趋势。

2.1.3　红线值

取自 2025.01–2026.05 服务良好期的历史波动下沿/上沿。地板线,非目标线——达标即维持现状,无需追求更高。

业务级不为每个业务单设绝对值(无法维护),主要看它自己和自己历史比的趋势,品类红线只作为应达到的底线参考。

2.1.4　趋势线阈值

红线防"跌穿地板",趋势线防"缓慢下滑"。两者并用。

状态触发条件(主要指标)处置
黄灯连续 3 周环比下滑　或　单周环比跌幅 > 5%进周报,排查,无需升级
红灯连续 5 周下滑　或　跌破红线升级:品类级 48h 出整改方案;平台级到管理者
绿灯以上均未触发周报一行汇总

2.1.5　业务分档处置标准

小业务卖家维护动力弱、服务指标常不理想,但单量小不影响品类/平台数据。按"未来会不会大"分三档。

档位是谁约束服务侧动作
主力单量大、已计入品类/平台数据硬约束:达品类红线破红线 → 48h 整改方案
增长苗子单量小但重点培育/自然快增软约束:趋势不许恶化入名单即提前对卖家提要求;趋势恶化即排查
长尾天生小、无增长预期仅防爆雷底线不管日常;单量起来自动升档
核心:对增长苗子提前管

对增长苗子,服务是它做大的入场资格。等单量起来污染品类数据再补救,口碑已先于规模崩坏。所以增长苗子虽单量小,服务也要提前要求;长尾只兜爆雷底线,几乎零维护。

增长苗子名单进出: ① 战略入口——第二/三段明确列为重点的业务自动进,由管理者圈定;② 数据入口——系统识别"连续 N 月自然增速 > X%"的业务,阈值由 BI 设定;③ 自动升档——长尾单量增长到接近计入品类数据时自动升为增长苗子。

2.2

低价竞争力标准

低价是 Coins 业务的核心增长因子之一,与服务同等重要。仅监测已具备价格优势的指定业务;未实现低价的业务作为第二层增长方向,不在本手册。

监测范围(4 个业务)

Dofus 金币　·　WOW TBC 金币　·　FC 金币　·　Dofus Touch 金币。这些业务已通过 订单数 × 卖家数 × 平台竞争排序规则 形成价格优势,监测目标是守住优势不滑坡,而非开拓新优势。

监测指标

数据来源:技术每日爬虫,追踪我方与 5 个竞品的价格对比、价格排名、价差优势比。

指标类型说明红线
价差优势比主我方价格 vs 竞品最低价的差距百分比。比排名信息量更大——反映"差距有多大",而非只看排第几待 BI 填入
价格排名辅在 6 家中的排位(我方常态为前三,非第一)跌出前三 → 黄灯

价差优势比红线取各业务 2025–26 历史优势区间的下沿(与最低价差距拉大超过该下沿则报警)。具体数值待价格竞争数据导出后由 BI 填入。

2.4

复购监测标准

复购是辅助观察指标,不设红线、只监测。它异常时用来反查是需求还是服务出了问题。

看绝对数,不看率

复购率上升可能只是因为边缘用户流失、只剩高频用户。所以主指标是复购买家绝对数,不是复购率。

状态触发动作
下钻归因复购买家绝对数连续 3 周下滑下钻:是需求还是服务问题
上报连续 4 周下滑　或　单月环比跌幅 > 10%下钻结论上报管理者
PART 03
执行动作 SOP
这一部分只讲"怎么操作":流程、步骤、频率。判断标准见第二部分。
3.1

服务履约大 SOP(两层联动)

服务监控分两层:第一层卖家治理(抓坏卖家,已有机制)+ 第二层数据监控(抓坏趋势,新建)。第一层处理"已知坏卖家",第二层发现"未知坏趋势"再反查出坏卖家或坏原因,第二层下钻结论大量回流为第一层处置输入。

3.1.1　每日卖家服务治理 SOP

针对单个卖家,日常由 2 名卖家管理客服处理(命中量不大,可与客服日常并行)。

  1. 命中报警
    卖家服务数据破规则红线 → 机器人自动报警,命中风险卖家(与数据监控分开的推送通道)。
  2. 人工核实
    客服确认是否确属服务差,排除买家原因、小基数误判。核实中发现的可固化逻辑,沉淀为命中前置过滤规则(优化①)。
  3. 分级处置
    确属服务差 → 由轻到重:① 警告　② 限期整改+观察　③ 下架　④ 禁止售卖(可提现)　⑤ 封禁(不可提现)。不属实 → 豁免、数据修正。
  4. 效果回写
    处置后记录效果数据(优化③),进入复盘。

处置流程图

三处优化(怎么做与价值)

① 命中规则动态更迭　已在做
人工核实中发现可固化的判断逻辑(如基数过小不命中、买家恶意退款不计入),持续沉淀为前置过滤规则,减少机械核实。

② 限期整改 + 观察档　看板功能待建

  1. 触发
    命中、核实确属服务差,但属于值得保留的卖家(如大卖家,或问题可整改)→ 不直接下架,进此档。
  2. 整改窗口
    给明确整改要求 + 窗口期(如 7 天)+ 复查时点。窗口内卖家正常经营,被标记为观察状态。
  3. 到期复查
    达标 → 解除观察、释放回监控池;未达标 → 升级到下一级处置。

价值:给值得救的卖家改正机会,避免一刀切下架损失供给;时限与复查防止纵容。当前缺口:到期"自动释放回监控池"看板功能未实现,现靠人工记录,易漏放或忘记复查(待办 #2)。

③ 处置效果回写　新增
每次处置后,系统记录该卖家或所在业务处置后的服务数据变化。

这一项价值最大

本质是把主观裁决"上系统",补上效果追踪的空白。当前裁决靠人工经验,做完很少回头看效果。回写后:① 每个裁决有"做了→效果如何"的完整记录;② 能评估裁决是否有效、力度是否合适(如某类问题"警告"无效需直接整改,或某处置过重导致优质卖家流失),处置规则可基于证据迭代;③ 反哺优化①,暴露哪些命中误判、哪些处置无效。

三项如何串联

回写(③)的效果数据,一方面验证整改档(②)是否真的让卖家改善,另一方面反哺规则更迭(①)。三者形成"执法 → 记录效果 → 评估有效性 → 优化规则与处置"的闭环。

3.1.2　每日 / 每周数据监控 SOP

针对业务/品类/平台三级数据本身,不针对单个卖家,补上第一层的盲区。

这一层补的盲区

第一层只看卖家级,三种问题会漏过去:① 多个卖家各自略差(均未命中)累加拖垮业务;② 好卖家走、中等卖家替代,无单个卖家变差但业务整体下台阶;③ 订单结构变化(大额占比升→退款率升),无人做错但数据恶化。

每日动作(机器人辅助,轻)

  • 三级指标自动刷新,红线 + 趋势线双重扫描。
  • 独立载体「每日业务服务健康度监控」只推异常项;无异常则一条绿灯汇总。
  • 此推送与卖家命中报警分开两个通道,避免被淹没。

每周动作(负责人核心职能:从被动到主动)

过去靠管理者每周亲自盯数据、发现后指派;现在由每日监控自动沉淀异常,负责人主动对当周亮灯业务下钻。

  1. 归因
    当周亮黄/红业务,定位是哪几个卖家拖的?属上述三种问题中的哪一种?
  2. 分诊
    卖家问题 → 回流第一层处置(哪怕未命中单卖家红线,纳入限期整改+观察);非卖家问题(订单结构/产品/需求)→ 升级或转对应负责人。
  3. 出周报
    本周亮灯项、归因结论、已采取/建议动作、需升级事项(见 §4.2)。
完整案例

某 Accounts 业务好评率周环比连跌 → 趋势线亮黄,但下面无卖家命中 → 周下钻发现 3 个卖家服务缓慢变差、各自未到红线但合起来拖垮业务 → 判定卖家问题 → 纳入限期整改+观察 → 窗口后复查:改善则解除,未改善则升级,回写复盘。

3.2

低价监测 SOP

4 个低价业务(Dofus / WOW TBC / FC / Dofus Touch 金币),与服务共用同一套报警机制和推送通道,运营无需学新逻辑。

  1. 每日扫描
    爬虫数据更新后,自动扫描价差优势比(主)与价格排名(辅)。
  2. 亮灯
    价差优势比触发趋势线(连续 3 周恶化)或排名跌出前三 → 黄灯;价差优势比跌破红线或连续 5 周恶化 → 红灯升级。
  3. 下钻
    优势比恶化时依次查:① 订单数是否下降　② 在售卖家数是否减少。到订单/卖家层即可,排序规则层转技术。
3.3

游戏动态 SOP

两项动作,篇幅从简:

  1. 日历维护与校准
    持续更新实际更新记录日历;用实际记录复盘年初推测的准确度,迭代预判能力。
  2. 每周订单量归因
    分析本周订单数增减,对照日历判断是否游戏动态导致。必须有基础数据依托:若与流量相关,进一步定位是否 UV 下降、是哪个渠道的 UV 下降,不凭主观判断。
3.4

复购监测 SOP

只监测,动作简单:

  1. 每周看绝对数
    观测复购买家绝对数(不看率)。
  2. 下钻归因
    连续 3 周下滑 → 下钻:同期游戏动态/订单量也降则为需求问题(转 §3.3);需求未降但服务指标恶化则为服务问题(回 §3.1)。
  3. 上报
    连续 4 周下滑或单月跌幅 > 10% → 下钻结论上报管理者。

4.2

协作机制:周报与升级

周报是管理者了解基本盘的唯一入口。绿灯翻篇、黄灯知悉、红灯升级。

4.2.1　周报结构

板块内容
① 平台总览平台级 6 指标红绿灯一行 + 本周状态
② 亮灯清单亮黄/红的品类与业务:指标、当前值 vs 红线、趋势
③ 下钻归因每个亮灯项的归因结论 + 分诊去向
④ 已采取动作本周卖家处置、纳入观察、转交事项
⑤ 升级项需管理者决策的事项
⑥ 低价竞争力4 个低价业务的价差优势比 / 排名变化(附竞品对比)
⑦ 游戏动态 + 订单归因TOP10 日历临近节点 + 本周订单量变化原因(附数据)

4.2.2　升级到管理者的条件

触发升级动作
平台级任一主指标破红线当周升级,负责人给止血方案
任一品类连续 4 周趋势恶化升级,出整改方案
低价业务价差优势比破红线 / 持续恶化升级,下钻订单/卖家并出方案
复购绝对数破阈值且归因指向服务升级(指向需求则转 §3.3)
增长苗子服务持续恶化、危及增长升级,提前介入
管理者角色

从"每周亲自盯数据的发起人",转为"只看周报结论、只处理升级项的验收人"。绿灯黄灯不惊动,只有红灯升级。这是从基本盘抽身、把精力转向增长的保证。
