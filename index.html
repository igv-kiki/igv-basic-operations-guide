iGV 基本盘服务运营手册

服务履约是基本盘的核心。本手册规定:为维持服务水平,每天每周做什么、什么时候报警、亮灯了怎么处理、谁负责。只讲执行,不展开论证。

红线的定位

所有红线数据是基于 2025–26 年服务良好期的历史数据,是"不滑坡"的地板线,不是"做更高"的目标线。达标即维持现状。

01

监控范围:守什么、不守什么

判断要不要重点守一件事,标准是:这件事停了,用户会不会走。据此,监控对象分三类、三种处理方式。

对象定位处理方式
服务履约 核心监控 设红线 + 趋势线,日常监控、亮灯处置。基本盘维持的精力主要投在这里。服务做差用户一定走;服务是平台级信任,任一业务出问题都会影响用户对整个平台的信任,所以全业务覆盖。
复购数据 辅助观察 不设红线。复购是"用户有需求 + 服务满意"后自然产生的结果,作为观察指标:它异常时,用来反查是需求还是服务出了问题(§10)。
游戏动态 外部参考 不可控,只追踪。用于解释订单量变化的原因、辅助判断(§9)。
02

指标与三层结构

2.1　指标分组

指标类型含义处置
好评率主要用户满意、愿意再来的直接结果硬红线 · 破线升级
递送完成率主要订单是否最终送达硬红线 · 破线升级
准时递送率主要是否在承诺时间内送达硬红线 · 破线升级
订单退款率主要按笔数,用户遇到问题的频率硬红线 · 破线升级
售后发起率次要用户主动发起售后的比例观察线 · 只提示
金额退款率次要按 GMV,退款的损失规模观察线 · 只提示

2.2　三层结构

层级看什么报警逻辑
平台级全业务汇总 6 指标绝对红线
品类级ACC / Coins 各自 6 指标(其他品类暂不设)各自绝对红线 + 趋势线
业务级TOP10 业务、增长苗子各自业务自比趋势线为主 + 套所属品类红线
分品类设红线的原因

品类服务特性不同,绝对值不可横向比。Accounts 售后率天然偏高、好评率天然偏低(账号交易存在封号、找回等问题);Items 准时率偏低(未必准时,但最终能送达,且不会出现封号找回等问题,因此差评和争议概率低)。所以红线各设各的,判断好坏看各自趋势。

03

红线值

取自 2025.01–2026.05 服务良好期的历史波动下沿/上沿。地板线,非目标线。

3.1　平台级红线

指标类型红线跌破处理
好评率主≥ 90%升级到管理者
递送完成率主≥ 96%升级到管理者
准时递送率主≥ 85%升级到管理者
订单退款率主≤ 5%升级到管理者
金额退款率次⚠ 8%只提示
售后发起率次⚠ 17%只提示

3.2　品类级红线(仅 Accounts 与 Coins)

指标类型AccountsCoins
好评率主≥ 87%≥ 87%
递送完成率主≥ 98%≥ 96%
准时递送率主≥ 98%≥ 85%
订单退款率主≤ 5%≤ 4%
金额退款率次⚠ 8%⚠ 5%
售后发起率次⚠ 20%⚠ 10%

其他品类订单量少,暂不单设红线;订单量增长后,以平台级红线作为参考值。

3.3　TOP10 业务与增长苗子红线

业务类型红线设定个体监控
TOP10 业务套用所属品类红线(ACC 业务用 ACC 红线、Coins 业务用 Coins 红线)业务自比趋势线(更重要)
增长苗子若为新品类,套用平台级红线;新品类如有特殊情况,届时再单设该品类红线业务自比趋势线
业务级以趋势线为主

业务级不追求每个业务单独设一套绝对值(无法维护)。判断单个业务服务好坏,主要看它自己和自己历史比的趋势;品类红线只作为"应达到的底线"参考。

3.4　长尾业务爆雷线(只兜底)

极小业务(月成交少)服务波动大,不管日常,只设一条所有业务不可逾越的声誉底线:

指标爆雷线触发动作
递送完成率< 85%大面积不发货风险,负责人介入
订单退款率> 15%退款失控风险
好评率< 75%口碑崩坏风险
04

趋势线阈值

红线防"跌穿地板",趋势线防"缓慢下滑"。两者并用。

状态触发条件(主要指标)处置
黄灯连续 3 周环比下滑　或　单周环比跌幅 > 5%进周报,负责人/客服排查,无需升级
红灯连续 5 周下滑　或　跌破红线升级:品类级 48h 出整改方案;平台级到管理者
绿灯以上均未触发周报一行汇总

次要指标仅在周报附表浮现,不参与升级。

05

业务分档处置:按潜力而非单量

小业务卖家维护动力弱、服务指标常不理想,但单量小不影响品类/平台数据。处理原则:按"未来会不会大"分三档。

档位是谁约束服务侧动作
主力单量大、已计入品类/平台数据硬约束:达品类红线破红线 → 48h 整改方案
增长苗子单量小但重点培育/自然快增软约束:趋势不许恶化入名单即提前对卖家提要求;趋势恶化即排查
长尾天生小、无增长预期仅防爆雷底线不管日常;单量起来自动升档
核心:对增长苗子提前管

对增长苗子,服务是它做大的入场资格。等单量起来污染品类数据再补救,口碑已先于规模崩坏。所以增长苗子虽单量小,服务也要提前要求;长尾只兜爆雷底线,几乎零维护。

增长苗子名单进出

  • 入口 A(战略):第二段(漏点修复)/ 第三段(新业务)明确列为重点的业务,自动进。由管理者圈定。
  • 入口 B(数据):系统识别"连续 N 月自然增速 > X%"的业务,提示纳入。阈值由 BI 设定。
  • 自动升档:长尾业务单量增长到接近计入品类数据时,自动升为增长苗子。

07

第二层 · 数据监控(每日 / 每周)

针对业务/品类/平台三级数据本身,不针对单个卖家。新建层,补上第一层的盲区。

为什么需要这一层

第一层只看卖家级,有三种问题漏过去:① 多个卖家各自略差(均未命中)累加拖垮业务;② 好卖家走、中等卖家替代,无单个卖家变差但业务整体下台阶;③ 订单结构变化(大额占比升 → 退款率升),无人做错但数据恶化。这些只能在业务级看到。

7.1　每日动作(机器人辅助,轻)

  • 三级指标自动刷新,红线 + 趋势线双重扫描。
  • 新建独立载体 「每日业务服务健康度监控」,只推异常项;无异常则一条绿灯汇总。
  • 此推送与卖家命中报警分开两个通道,避免被淹没。

7.2　每周动作(负责人核心职能:从被动到主动)

过去靠管理者每周亲自盯数据、发现后指派;现在由每日监控自动沉淀异常,负责人主动对当周亮灯业务下钻。

  1. 归因
    当周亮黄/红业务,定位是哪几个卖家拖的?属上述三种问题中的哪一种?
  2. 分诊
    归因到卖家问题 → 回流第一层处置(哪怕未命中单卖家红线,纳入限期整改+观察);归因到非卖家问题(订单结构/产品/需求)→ 升级或转对应负责人。
  3. 出周报
    本周亮灯项、归因结论、已采取/建议动作、需升级事项。
08

两层联动

第一层处理"已知坏卖家",第二层发现"未知坏趋势"再反查出坏卖家或坏原因。第二层的下钻结论,大量回流为第一层的处置输入。

完整案例

某 Accounts 业务好评率周环比连跌 → 第二层趋势线亮黄,但下面无卖家命中 → 周下钻发现 3 个卖家服务缓慢变差、各自未到红线但合起来拖垮业务 → 判定卖家问题 → 纳入限期整改+观察 → 窗口后复查:改善则解除,未改善则升级,回写复盘。

游戏动态追踪

游戏热度不可控,只追踪,用于解释订单量变化。范围是 TOP10 游戏业务。两项动作:

9.1　日历维护

日历用途状态
年度推测日历年初预测本年度 TOP10 游戏更新节点已有
实际更新记录日历记录实际发生的版本/活动更新点已有
推测 vs 实际校准用实际记录复盘推测准确度,迭代预判能力新增

9.2　每周订单量变化归因

每周分析订单数的增加或下降,并定位原因,必须有基础数据依托:

  1. 订单数变化
    本周订单数环比上升还是下降?
  2. 是否游戏动态导致
    对照日历,是否有版本更新或热度变化对应?
  3. 数据依托验证
    若与流量相关,需进一步定位:是否 UV 下降?是哪个渠道的 UV 下降?给出基础数据,不凭主观判断。
10

复购监测

复购是辅助观察指标,不设红线。三项动作:观察数据、下钻分析、建立用户忠诚度计划。

10.1　观察指标与阈值

看绝对数,不看率

复购率上升可能只是因为边缘用户流失、只剩高频用户。所以主指标是复购买家绝对数,不是复购率。

状态触发动作
下钻归因复购买家绝对数连续 3 周下滑下钻:是需求(游戏动态)还是服务问题
上报连续 4 周下滑　或　单月环比跌幅 > 10%下钻结论上报管理者

10.2　下钻归因方向

  • 同期游戏动态/订单量也在降 → 需求侧问题,转 §9 分析。
  • 需求未降但服务指标也在恶化 → 服务侧问题,回 §3/§7 处理。

10.3　用户忠诚度计划

负责人交付方案,产研实现

服务运营负责人产出用户忠诚度计划的大致方案(目标用户、权益/召回形式、触发逻辑框架),具体实现由产研部门负责。此项为方案交付,不设跟踪指标。

12

周报与升级

周报是管理者了解基本盘的唯一入口。绿灯翻篇、黄灯知悉、红灯升级。

12.1　周报结构

板块内容
① 平台总览平台级 6 指标红绿灯一行 + 本周状态
② 亮灯清单亮黄/红的品类与业务:指标、当前值 vs 红线、趋势
③ 下钻归因每个亮灯项的归因结论 + 分诊去向
④ 已采取动作本周卖家处置、纳入观察、转交事项
⑤ 升级项需管理者决策的事项
⑥ 游戏动态 + 订单归因TOP10 日历临近节点 + 本周订单量变化原因(附数据)

12.2　升级到管理者的条件

触发升级动作
平台级任一主指标破红线当周升级,负责人给止血方案
任一品类连续 4 周趋势恶化升级,出整改方案
复购绝对数破阈值且归因指向服务升级(指向需求则转 §9)
增长苗子服务持续恶化、危及增长升级,提前介入
管理者角色

从"每周亲自盯数据的发起人",转为"只看周报结论、只处理升级项的验收人"。绿灯黄灯不惊动,只有红灯升级。这是从基本盘抽身、把精力转向增长的保证。
